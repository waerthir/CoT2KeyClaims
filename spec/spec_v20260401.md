# CoT2KeyClaims Task Spec

## 1. 任务定位

本任务在逻辑上是一个明确分层的三层处理流程：

1. 原始题目生信息 -> 结构化题目信息
2. 结构化题目信息 -> CoT
3. CoT -> key claims

在文件组织上，还额外包含一层输入存放目录：

- `layer_input`

本 spec 的目标是：

- 定义三层模块的职责边界
- 定义每层的输入输出
- 定义文件级组织方式
- 定义第二层、第三层“先碎片、后打包”的保存原则
- 定义第三层中 `key claims` 的粒度与标签体系

本 spec 当前不讨论：

- 具体代码实现
- 具体模型 prompt
- 具体调度与并发

## 2. 关于 example 的使用边界

现有 example 只用于帮助理解 `key claims` 的粒度，不用于规定真实数据的组织方式。

因此：

- example 不是 schema 模板
- example 的字段层级不直接照搬
- example 的 type 设计不直接继承

本 spec 只吸收 example 的一个核心结论：

- `key claims` 应是到达正确答案所必需的最小骨架，而不是完整 CoT

## 3. 标识规则

### 3.1 `problem_id`

- 每道题的唯一标识使用 `problem_id`
- `problem_id` 是字符串字段
- 从长期设计意图看，`problem_id` 最好由上游或外部数据直接提供
- 第一层的理想职责是继承并保留外部给定的 `problem_id`
- 当前版本中，`problem_id` 由第一层 `Raw2Problems` 临时生成，这只是当前阶段的权宜实现
- 第二层与第三层只继承，不重新生成
- 当前第一层实现中，`problem_id` 的目标格式固定为：`prob_` + 24 位小写 hex
- 示例：`prob_9b0a7ef70ee83d702776eea6`

当前第一层实现中，`problem_id` 的生成职责归属于 `ProblemIdGenerator`。

当前记录的实现级生成逻辑是：

1. 取 `file_id`
2. 取当前题目的 `record_index`
3. 取原始字段中的 `picture`
4. 取原始字段中的 `question`
5. 取原始字段中的 `standard_answer`，若不存在则回退取 `answer`
6. 将上述内容组成字典
7. 使用 `json.dumps(..., ensure_ascii=False, sort_keys=True)` 转成字符串
8. 对该字符串做 `sha1`
9. 截取前 24 位小写 hex
10. 最前面加上 `prob_`

但需要明确：

- 这套生成逻辑只是当前第一层实现中的过渡方案
- 它不代表长期架构中 `problem_id` 必须由第一层自行生成
- 一旦上游开始稳定提供 `problem_id`，第一层应优先直接继承外部给定值
- 到那时，第二层与第三层的依赖关系不变，只是第一层不再承担生成职责

当前不再单独引入：

- `problem_info_id`
- 任何额外题目级哈希 id

### 3.2 `file_id`

- `file_id` 是字符串字段
- 当前版本中，`file_id` 直接取自工作单元目录名
- 包级对象统一使用 `file_id` 作为文件级标识

例如：

- 工作单元目录名为 `aaa`
- 第一层主包文件 id 为 `aaa`
- 第二层主包文件 id 为 `aaa`
- 第三层主包文件 id 为 `aaa`

### 3.3 `method_id`

- `method_id` 是字符串字段
- `method_id` 由 `problem_id` 与分配的序数 id 用下划线连接构成
- 序数 id 为整数
- 序数 id 在单个 `problem_id` 内部从 `1` 开始递增
- 一旦在第二层分配完成，第三层必须原样继承，不再改号

示例：

- 若 `problem_id = prob_9b0a7ef70ee83d702776eea6`
- 且某题的第一个方法序数 id 为 `1`
- 则 `method_id = prob_9b0a7ef70ee83d702776eea6_1`

注意：

- 这里使用的是当前第一层实现中的 `problem_id` 示例格式
- 若未来第一层更改 `problem_id` 规则，`method_id` 的前缀部分也随之变化

因此，`method_id` 本身就是方法及对应碎片的唯一标识。

### 3.4 碎片命名规则

在当前架构中，不再为碎片额外生成独立 id。

碎片文件名规则固定为：

- `method_id + ".json"`

示例：

- 若 `method_id = prob_9b0a7ef70ee83d702776eea6_1`
- 则碎片文件名为 `prob_9b0a7ef70ee83d702776eea6_1.json`

这条规则对第二层和第三层完全一致。

## 4. 文件组织

### 4.1 顶层目录

当前采用四层目录组织：

- `layer_input`
- `layer_problem`
- `layer_cot`
- `layer_key_claims`

其中：

- `layer_input` 保存原始输入工作单元
- `layer_problem` 保存第一层输出主包
- `layer_cot` 保存第二层输出主包与 CoT 碎片
- `layer_key_claims` 保存第三层输出主包与 key claims 碎片

### 4.2 以工作单元目录名为工作单元

假设某个原始题目工作单元名为 `aaa`，则四个 layer 下都创建同名工作目录：

- `layer_input/aaa/`
- `layer_problem/aaa/`
- `layer_cot/aaa/`
- `layer_key_claims/aaa/`

这里的 `aaa` 是当前这一批数据对应的稳定工作单元目录名。

当前第一层实现中，默认会遍历 `layer_input/` 下的所有一级子目录，并将每个子目录都视为一个待处理工作单元。

当前第一层实现中的 `list_work_units()` 会：

1. 遍历 `layer_input/` 下的候选工作单元目录
2. 为每个目录构建一个 `WorkUnit`
3. 返回一个有序的 `list[WorkUnit]`

这里的 `WorkUnit` 是运行期对象，不是输出 JSON 的字段。它用于保存：

- `file_id`
- `input_dir`
- `input_file`
- `image_dir`
- `output_dir`
- `output_file`

### 4.3 每层目录的内容

输入层：

- `layer_input/aaa/` 下存放当前工作单元的原始题目 JSON
- `layer_input/aaa/` 下还应包含一个图片目录

关于图片目录：

- 当前确认“必须存在”
- 当前第一层实现统一使用 `test/`
- 因此第一层标准化后的 `images` 字段中，每个元素都必须写成相对于工作单元根目录 `aaa/` 的相对路径，例如 `test/img1.png`
- 未来设计上允许不同工作单元使用不同的图片目录名
- 但当前代码尚未实现“按工作单元分别指定图片目录名”的机制

第一层主包层：

- `layer_problem/aaa/` 下存放这一批题目结构化后的主包 JSON 文件

第二层：

- `layer_cot/aaa/` 下存放这一批题目的 CoT 主包 JSON 文件
- `layer_cot/aaa/CoT_fragments/` 下存放 CoT 碎片

第三层：

- `layer_key_claims/aaa/` 下存放这一批题目的 key claims 主包 JSON 文件
- `layer_key_claims/aaa/key_claims_fragments/` 下存放 key claims 碎片

### 4.4 主文件命名

每层的主文件直接使用工作单元名作为文件 id。

推荐形式：

- `layer_input/aaa/<single_input_file>.json`
- `layer_problem/aaa/aaa.json`
- `layer_cot/aaa/aaa.json`
- `layer_key_claims/aaa/aaa.json`

## 5. 第一层：生信息 -> 题目信息

### 5.1 目标

第一层负责把来源不定、格式未定的原始题目生信息，整理成统一的题目信息主包。

当前代码对应关系是：

- `config.py`：保存第一层配置项
- `raw2problems.py`：保存第一层完整逻辑
- 运行入口：`python raw2problems.py`

关于 `problem_id` 的职责定位，需要区分“当前实现”和“长期目标”：

- 当前实现：第一层临时生成 `problem_id`
- 长期目标：上游或外部数据直接给好 `problem_id`，第一层只做继承和标准化

这一层只处理题目本体，不负责：

- 生成 CoT
- 提取 claims
- 判断 claim 粒度

### 5.2 输入

第一层当前按工作单元处理输入。

默认情况下，它会遍历 `layer_input/` 下的所有一级子目录，并处理每个工作单元。

单个工作单元的输入示意为：

- `layer_input/aaa/aaa.json`

同时，`layer_input/aaa/` 下还应有图片目录：

- `layer_input/aaa/test/`

当前第一层实现要求：

- 每个工作单元目录中只允许存在一个可读取的 `*.json`
- 若找不到输入 `json`，则报错
- 若找到多个输入 `json`，也报错

当前已确认的一版原始输入 schema 是：

```json
[
  {
    "picture": "img1.png",
    "question": "题目文本",
    "standard_answer": "标准答案"
  }
]
```

说明：

- 顶层是 `[]`
- 列表中的每个元素是一个 `{}` 题目对象
- 当前第一层实现要求每个元素至少有：
  - `picture`
  - `question`
  - `standard_answer` 或 `answer`

当前第一层实现的兼容行为：

- 若 `standard_answer` 缺失但存在 `answer`，则回退使用 `answer`
- 若 `picture` 是字符串，则输出单元素 `images` 列表
- 若 `picture` 是字符串列表，则逐个改写路径
- 若 `picture` 是 `null`，则当前配置下输出 `images = []`

### 5.3 输出

第一层输出是一个主包 JSON 文件，可称为：

- `ProblemPackage`

其文件位置示意为：

- `layer_problem/aaa/aaa.json`

### 5.4 `ProblemPackage` 结构

```json
{
  "file_id": "aaa",
  "stage": "raw_to_problem",
  "source_file_name": "actual_input_file.json",
  "problems": [
    {
      "problem_id": "prob_9b0a7ef70ee83d702776eea6",
      "question_text": "题目文本",
      "standard_answer": "标准答案",
      "images": ["test/img1.png"],
      "source_meta": {},
      "multi_solution_hint": null,
      "ingest_status": "ready"
    }
  ]
}
```

### 5.5 第一层设计说明

这里不使用 `problem_info_id`，而是直接在 `problems` 数组中以 `problem_id` 作为题目主键。

理由是：

- 当前架构下，第一层已经负责生成稳定的题目级标识
- 第二层真正消费的是“一个问题集合文件”
- `file_id` 足以标识这一批数据，`problem_id` 足以标识批内单题

但这只是当前实现状态。

从长期设计意图看：

- 第一层最好不要永久承担 `problem_id` 生成职责
- 更理想的方式是上游直接把 `problem_id` 提供好
- 第一层只负责读取、保留、标准化，并把它传给第二层和第三层

当前第一层实现中的默认输出值为：

- `source_meta = {}`
- `multi_solution_hint = null`
- `ingest_status = "ready"`

当前第一层实现中的 `ProblemPackageWriter` 会：

- 使用 `utf-8`
- 使用 `indent = 2`
- 使用 `ensure_ascii = False`
- 在文件末尾补一个换行

## 6. 第二层：题目信息 -> CoT

### 6.1 目标

第二层负责读取第一层的 `ProblemPackage`，调用模型生成 CoT，并将结果组织成：

- 可追踪的 CoT 碎片
- 可供第三层消费的 CoT 主包

第二层必须遵循：

1. 先生成 CoT 碎片
2. 每个碎片生成后立即保存
3. 碎片收齐后再打包成主 JSON

### 6.2 输入

第二层的标准输入是第一层的主包文件：

- `layer_problem/aaa/aaa.json`

第二层不直接读取原始脏数据。

### 6.3 第二层碎片：`CoTFragment`

第二层的最小保存单位是 CoT 碎片。

每个碎片对应：

- 某个 `problem_id`
- 某个 `method_id`
- 某次已保存的 CoT 结果

其文件位置示意为：

- `layer_cot/aaa/CoT_fragments/<method_id>.json`

推荐逻辑结构：

```json
{
  "file_id": "aaa",
  "problem_id": "prob_9b0a7ef70ee83d702776eea6",
  "method_id": "prob_9b0a7ef70ee83d702776eea6_1",
  "cot_steps": [
    "步骤1",
    "步骤2",
    "步骤3"
  ],
  "generated_answer": "模型答案",
  "standard_answer": "标准答案",
  "answer_match_status": "matched",
  "validation_status": "accepted",
  "ready_for_packaging": true
}
```

### 6.4 第二层关于 `method_id` 的规定

第二层负责给每道题的方法分配 `method_id`。

规定如下：

- `method_id` 是字符串字段
- `method_id = problem_id + "_" + 序数id`
- 序数 id 从 `1` 开始编号
- 序数 id 在同一 `problem_id` 内部递增
- 一旦写入碎片文件，不再改动

这意味着：

- 第二层是 `method_id` 的源头
- 第三层只能继承，不能重排

### 6.5 第二层保存原则

第二层必须“生成即保存”：

- 每产生一个有效 CoT，就先落成一个 `CoTFragment`
- 不等待整批题跑完再统一写出
- 最终主包只能从已有碎片组装

### 6.6 第二层主包：`CoTPackage`

第二层最终输出一个主包 JSON 文件：

- `layer_cot/aaa/aaa.json`

这个文件的 id 直接就是：

- `aaa`

需要明确的是：

- 这个主 JSON 文件的内容不是“单纯的题目文本”
- 而是第二层最终汇总后的 CoT package
- 它是第三层的标准输入

推荐逻辑结构：

```json
{
  "file_id": "aaa",
  "stage": "problem_to_cot",
  "source_problem_file_id": "aaa",
  "problems": [
    {
      "problem_id": "prob_9b0a7ef70ee83d702776eea6",
      "question_text": "题目文本",
      "standard_answer": "标准答案",
      "images": ["test/img1.png"],
      "multi_solution_hint": null,
      "cot_methods": [
        {
          "method_id": "prob_9b0a7ef70ee83d702776eea6_1",
          "cot_steps": ["步骤1", "步骤2"],
          "generated_answer": "答案A",
          "answer_match_status": "matched",
          "validation_status": "accepted"
        }
      ]
    }
  ]
}
```

### 6.7 第二层打包条件

第二层不应在碎片明显缺失时强行打包。

至少要满足：

- 主包中每道题都有 `problem_id`
- 被纳入主包的每个方法都已有对应 CoT 碎片
- 对应 CoT 已完成当前批次要求的基础校验

## 7. 第三层：CoT -> key claims

### 7.1 目标

第三层负责读取第二层的 `CoTPackage`，将其中每个方法的 CoT 拆成 `key claims`，并组织成：

- claims 碎片
- claims 主包

第三层同样遵循：

1. 先逐方法生成碎片
2. 每个碎片生成后立即保存
3. 碎片收齐后再打包主 JSON

### 7.2 输入

第三层的标准输入是第二层主包文件：

- `layer_cot/aaa/aaa.json`

第三层不直接以零散 CoT 片段为主输入。

### 7.3 第三层碎片：`ClaimFragment`

第三层的最小保存单位是 claims 碎片。

每个碎片对应：

- 某个 `problem_id`
- 某个 `method_id`
- 该方法下的一整组 key claims

其文件位置示意为：

- `layer_key_claims/aaa/key_claims_fragments/<method_id>.json`

推荐逻辑结构：

```json
{
  "file_id": "aaa",
  "problem_id": "prob_9b0a7ef70ee83d702776eea6",
  "method_id": "prob_9b0a7ef70ee83d702776eea6_1",
  "key_claims": [
    {
      "claim_id": "m1_c1",
      "text": "题干给出 r 组为 rhc1 功能缺失突变体。",
      "claim_type": "题目信息",
      "is_final_answer": false
    },
    {
      "claim_id": "m1_c2",
      "text": "图2中高浓度 CO2 条件下 r 组气孔开放度高于 wt 组。",
      "claim_type": "图片信息",
      "is_final_answer": false
    },
    {
      "claim_id": "m1_c3",
      "text": "rhc1 基因产物促进气孔关闭。",
      "claim_type": "推理信息",
      "is_final_answer": true
    }
  ],
  "extraction_status": "accepted",
  "ready_for_packaging": true
}
```

### 7.4 第三层保存原则

第三层也必须“生成即保存”：

- 每完成一个方法的 claims 提取，就立刻保存为 `ClaimFragment`
- 不等待整批题结束后再统一写出
- 最终主包只从已保存碎片汇总

### 7.5 第三层主包：`ClaimsPackage`

第三层最终输出主包文件：

- `layer_key_claims/aaa/aaa.json`

这个文件的 id 直接就是：

- `aaa`

需要明确的是：

- 这个主 JSON 文件的内容不是“题目原文”
- 而是第三层最终汇总后的 key claims package
- 它是 claims 层的标准结果文件

推荐逻辑结构：

```json
{
  "file_id": "aaa",
  "stage": "cot_to_claims",
  "source_cot_file_id": "aaa",
  "problems": [
    {
      "problem_id": "prob_9b0a7ef70ee83d702776eea6",
      "question_text": "题目文本",
      "standard_answer": "标准答案",
      "images": ["test/img1.png"],
      "cot_methods": [
        {
          "method_id": "prob_9b0a7ef70ee83d702776eea6_1",
          "cot_steps": ["步骤1", "步骤2"],
          "key_claims": [
            {
              "claim_id": "m1_c1",
              "text": "图中给出三期岸线变化。",
              "claim_type": "图片信息",
              "is_final_answer": false
            },
            {
              "claim_id": "m1_c2",
              "text": "冬半年北侧侵蚀、南侧淤积。",
              "claim_type": "推理信息",
              "is_final_answer": false
            },
            {
              "claim_id": "m1_c3",
              "text": "冬季移动固沙障应主要布设在北侧。",
              "claim_type": "推理信息",
              "is_final_answer": true
            }
          ]
        }
      ]
    }
  ]
}
```

### 7.6 第三层打包条件

第三层的完整性检查更明确，因为输入已经是第二层主包。

对第三层而言：

- 第二层主包中每个被纳入的 `method_id`
- 都必须对应一个且仅一个合格的 `ClaimFragment`

## 8. “先碎片，后打包”的统一原则

第二层和第三层都必须遵循同一套原则。

### 8.1 碎片优先

- 任何生成结果先写成碎片
- 主包只是汇总，不是原始生成入口

### 8.2 打包只读碎片

- 打包阶段只能读取已存在的碎片
- 不允许在打包阶段临时重新生成内容

### 8.3 碎片是事实记录，主包是传递单元

- 碎片用于留痕、补跑、回收、校验
- 主包用于作为下游标准输入

## 9. `key claim` 定义

一个 `key claim` 是：

> 为了从当前题目的输入到达正确答案，所必需的一条最小命题单元。

这里的“最小”不是越细越好，而是：

- 再压缩会丢失解题必需信息
- 再拆分会退化成机械中间式或零散碎片

## 10. `key claims` 的粒度规则

### 10.1 基本要求

每条 claim 应满足：

- 一条 claim 只表达一个可独立使用的命题
- 只保留对最终答案真正必要的信息
- 尽量写成最简表达
- 不与其他 claims 重复

### 10.2 允许的合并

“原子化”不等于“每个数都单独一条”。

以下情形允许作为一条 claim 保留：

- 同一处题干中绑定出现且必须一起使用的一组条件
- 同一局部图像中一次性读取的一组视觉事实
- 同一个紧密绑定的关系表达

### 10.3 不应单独保留的内容

以下内容通常不应成为 key claim：

- 纯机械代数变形的每个中间式
- 同义重复
- 修辞连接词
- 与答案无关的观察
- 模型自我确认、试错、回退痕迹

## 11. 标签体系

本任务只使用三类标签：

- `题目信息`
- `图片信息`
- `推理信息`

## 12. 最终答案与多解题规则

### 12.1 最终答案

- 每个 `method_id` 必须显式包含最终答案
- 最终答案 claim 的类型固定为 `推理信息`
- 最终答案默认位于该方法 claims 的末尾

### 12.2 多解题

- 一道题的不同解法必须拆成不同 `method_id`
- 每个 `method_id` 都拥有自己独立的 CoT 和 claims
- 不同方法不得混合成一条链

## 13. 自检标准

### 13.1 第二层自检

一个 CoT 能进入第二层主包，至少要满足：

- 有明确的 `problem_id`
- 有明确的 `method_id`
- 已保存为独立碎片
- 与标准答案的关系明确

### 13.2 第三层自检

一个 claims 结果能进入第三层主包，至少要满足：

- 有明确的 `problem_id` 和 `method_id`
- 每条 claim 的标签清晰
- 包含最终答案 claim

## 14. 当前已经确定的核心命名

- 题目统一字段名：`problem_id`
- 文件统一字段名：`file_id`
- 方法及碎片统一字段名：`method_id`
- `method_id = problem_id + "_" + 序数id`
- 碎片文件名统一采用 `method_id + ".json"`
